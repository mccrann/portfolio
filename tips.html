The clearest architecture rule

The agent should think in business tasks.

The application should think in implementation details.

Judgement on one side · Deterministic work on the other

The agent decides what should happen. The application makes it happen safely.

DocBot is given a small number of high-level business tools rather than a bag of low-level GitHub operations. That keeps the agent focused on the real question: does the customer documentation need to change, and why?

The application retains responsibility for predictable operations: authentication, validation, retries, fetching repository data, creating branches, committing files and opening pull requests.

It is almost the opposite of the usual AI metaphor. The agent acts like the human doing the higher-level thinking, while the application quietly handles the repetitive work.

A deeper thought

Software increasingly looks like a reconstruction of ourselves.

At the bottom, software has basic functions doing mundane, essential work: schedulers, data retrieval, recurring jobs and background processes.

Above that are simple decisions: if this happens, do that. Then comes comparison and pattern matching: do these dates overlap, does this event resemble an earlier one, is something out of place?

AI adds another layer: reasoning. To me, the stack maps surprisingly well onto human experience—heartbeat and breathing, bodily signals such as thirst and tiredness, subconscious pattern recognition, and finally the part of us that can think about what all of it means.

How that idea appears in DocBot

Three different kinds of responsibility.

The system works because the agent, the application and the reviewer are not pretending to be interchangeable.

01 · AGENT

Reason

Interpret the product change, compare it with the help content and decide whether customers need different information.

  • Customer impact
  • Documentation judgement
  • Evidence and rationale
02 · APPLICATION

Execute

Perform the repository work in a predictable way and reject output that does not match the expected structure.

  • GitHub authentication
  • Branches, files and commits
  • Validation and retries
03 · HUMAN

Own the outcome

Check the evidence, edit the wording and decide whether the customer-facing change is accurate and ready to publish.

  • Accuracy
  • Tone and context
  • Final approval

The DocBot flow

A reasoning layer inside a normal application.

The agent is one part of the system, not the whole system.

Next.js · GitHub capabilities · Structured output

The application surrounds the agent with controlled capabilities.

A user starts the workflow through the Next.js frontend. The application retrieves repository data and presents a small set of useful tools to the DocBot agent.

The agent returns a structured documentation decision. The application validates that decision, creates the documentation pull request and relies on Netlify previews and GitHub review for the final human check.

Good architecture compounds

Once the patterns exist, AI can add features surprisingly quickly.

Once the architecture, API patterns and workflow are in place, it becomes much easier for AI to add new features to a project.

It no longer has to invent how everything should work. It can follow the patterns that already exist, reuse the established boundaries and fill in the missing pieces.

I saw this when I added feedback retrieval to DocBot. The API routes, GitHub calls, authentication and error-handling patterns were already established, so adding another API call was straightforward.

The useful part was not that AI magically designed the feature. The useful part was that I had already designed a system clear enough for AI to extend without guessing.

A tiny practical tip

Repeated typing is still repetitive work.

TextExpander is an absolute must for me as a developer. It is not a good use of time—or attention—to type the same debugging statements over and over again.

For example, I can copy a variable name, type a short trigger such as ;cl, and expand it into the full console.log() statement.

It is not glamorous architecture, but removing tiny repeated interruptions makes it easier to stay focused on the actual problem.

Things that broke

The useful lessons were hiding inside the failures.

DocBot’s first working version was built through several problems that forced the architecture to become clearer.

01 · AUTHENTICATION

The wrong GitHub auth model

The app originally relied on a personal-access-token approach. That became unreliable for write operations such as creating refs and opening pull requests.

  • Moved to GitHub App authentication
  • Separated identity from my personal account
  • Reduced permissions to the capabilities required
02 · ENVIRONMENT

The private key format was wrong

The GitHub App private key needed to be stored as a single-line environment value with escaped newlines.

  • Multiline formatting caused signing failures
  • The error appeared as invalid key data
  • Environment representation mattered as much as the key itself
03 · NEXT.JS BOUNDARIES

Client code imported server-only logic

A client component directly imported GitHub code that depended on server-only modules, so Next.js attempted to bundle something that should never reach the browser.

  • Moved GitHub work behind server routes
  • Kept secrets on the server
  • Made frontend and application boundaries explicit

Prototype now, scale deliberately

Do not add a database simply because a larger system might need one.

For the prototype, GitHub is enough to track what DocBot has already processed. Pull-request links, labels, comments and review history provide a useful audit trail without creating another persistence layer.

At a larger scale, I would move workflow state, retries, feedback retrieval and reporting into a database. GitHub would still remain the source of truth for the actual documentation change and review evidence.

The important distinction is between what the prototype needs now and what a production system would need under different load, reliability and multi-organisation requirements.

Before adding cleverness

Test the boring parts first.

Memory and feedback loops are only useful if the underlying workflow can be trusted.

Unit tests · Workflow orchestration · GitHub capabilities

New intelligence should not make the basic workflow fragile.

Before adding memory, historical feedback and guidance generation, I added unit tests around orchestration and the GitHub capability layer.

That creates a safer foundation for experimenting with features that are less deterministic. The agent may reason differently as its guidance evolves, but branch creation, file validation and repository operations should remain dependable.
