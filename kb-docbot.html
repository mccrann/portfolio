AI-assisted documentation workflow

KB DocBot

KB DocBot analyses recently merged product pull requests, checks the customer help documentation and creates a documentation pull request when something needs to change.

I built it to explore a problem I had already seen in real product teams: the product changes, the code ships and the documentation quietly falls behind.

DocBot is designed to reduce that gap without allowing AI to publish customer-facing information unchecked.

The problem

Documentation updates are easy to miss because the information needed to identify them is spread across different places.

An engineer understands the code change. A technical writer understands the help content. Support teams understand where customers are becoming confused. But the connection between those things is often manual.

That means help documentation can remain technically correct while still describing behaviour that no longer exists.

I wanted to see whether an AI agent could help make that connection: not by publishing documentation itself, but by finding likely gaps, explaining its reasoning and preparing a useful first draft for a person to review.

What I built

KB DocBot connects three repositories and workflows:

  • A product repository containing recently merged code changes
  • A documentation repository containing the customer help content
  • A Next.js application that coordinates the GitHub and AI workflows

From the application, DocBot can:

  • Retrieve recently merged and unprocessed product pull requests
  • Read the files and code patches changed by each pull request
  • Fetch the current Markdown help documentation
  • Decide whether the change affects customers
  • Identify the documentation pages that need to change
  • Generate updated content and explain the evidence behind it
  • Create a branch, commit and documentation pull request
  • Label processed product pull requests to avoid duplicate work

How it works

  1. Fetch the product changes. DocBot retrieves merged pull requests from GitHub, including their titles, descriptions, changed files and code patches.
  2. Process each pull request separately. Each product pull request is analysed on its own so the agent can reason about one change at a time rather than trying to interpret a large collection of unrelated changes.
  3. Read the current help content. The agent checks the documentation repository for pages that describe the affected behaviour.
  4. Make a documentation decision. The agent either returns a no-change decision or proposes specific Markdown updates supported by evidence from the product change and existing documentation.
  5. Build one coherent documentation update. Product pull requests are processed in chronological merge order. Each new decision sees the documentation changes produced by earlier pull requests.
  6. Create a pull request for review. DocBot creates the GitHub branch, commit and documentation pull request. A person still decides whether it should be edited, approved or rejected.

Handling overlapping product changes

One of the more interesting problems was working out what should happen when several product pull requests affect the same area of the product.

Simply generating a complete documentation page for every pull request and trying to combine the results afterwards is unreliable. Two agents may return different versions of the same file even when their changes are not actually in conflict.

Instead, DocBot processes the product pull requests from oldest to newest and maintains a working copy of the documentation as it goes.

  • If two product pull requests change different behaviour, both documentation updates are preserved.
  • If a later product pull request changes the same behaviour again, the newest merged change takes precedence.
  • Unrelated updates already made to the same documentation page remain intact.

This creates one final version of the help content that reflects the actual order in which the product changed.

Human review is part of the architecture

I did not want DocBot to publish directly to the help centre.

AI can be useful for finding patterns, comparing code with existing content and producing a first draft. It should not silently decide what customers are told.

Every proposed change is therefore created as a normal GitHub pull request. The reviewer can inspect the evidence, compare the changes, edit the wording and decide whether the update is ready to merge.

This keeps accuracy, tone and release timing under human control while still removing much of the repetitive investigation and drafting work.

Building a feedback loop

Each documentation pull request includes a simple feedback prompt asking the reviewer to react with:

  • 👍 Good robot when the proposal is accurate and useful
  • 👎 Bad robot when it is incorrect, incomplete or unnecessary

Reviewers can also leave comments or edit the proposed documentation before merging it.

For the prototype, GitHub acts as a lightweight feedback and audit store. It already contains:

  • The original product change
  • DocBot’s proposed documentation
  • The evidence used to make the decision
  • Reviewer reactions and comments
  • Any human edits
  • The final merged or rejected result

That means I can add a meaningful feedback loop without introducing a database before the project actually needs one.

Using GitHub as lightweight memory

DocBot also uses GitHub labels as a simple form of workflow memory.

Once a product pull request has been analysed successfully, it can be labelled as processed. Future runs ignore those pull requests, which prevents duplicate documentation proposals without requiring a separate persistence layer.

This approach is deliberately small and transparent. The state of the workflow can be inspected directly in GitHub rather than hidden inside an application database.

Thinking beyond the prototype

For the prototype, GitHub is doing a lot of the heavy lifting. It stores the product pull requests, the documentation pull requests DocBot creates, reviewer comments, reactions, edits and the final merged content.

That gives me a lightweight audit trail without adding a database before I actually need one.

I do not think GitHub alone would be the best long-term solution for a company using this at scale, though.

In a production system, I would keep GitHub as the source of truth because it contains the actual evidence: the original code change, DocBot’s proposal, the reviewer’s feedback and the final documentation.

I would then use GitHub webhooks to send structured feedback into a database.

That would make it much easier to:

  • Retrieve relevant past decisions before analysing a new PR
  • Separate data between organisations and repositories
  • Track acceptance, rejection and edit rates
  • Identify the kinds of changes DocBot gets wrong most often
  • Search feedback by product area, file or documentation topic
  • Build organisation-specific guidance over time

The database would not replace GitHub. It would act as a faster, searchable operational layer on top of it. 

Product PR merged
        ↓
DocBot proposes a documentation update
        ↓
Reviewer approves, edits or rejects it in GitHub
        ↓
GitHub webhook sends the feedback to the application
        ↓
Feedback is stored and analysed
        ↓
Relevant examples and approved guidance are used in future runs

Controlled learning, not automatic self-editing

A future version could use an evaluation agent to analyse previous documentation pull requests and identify recurring feedback.

For example, it might discover that internal constant changes rarely need documentation, or that changes to payout timing usually affect a particular help page.

I would not allow the agent to rewrite its own instructions automatically.

Instead, the evaluation agent could propose updates to a version-controlled guidance file. A person would review and approve those changes before they affected future decisions.

This would allow the system to improve over time without removing human oversight or making its behaviour difficult to understand and audit.

What this project demonstrates

KB DocBot is not just a prompt connected to GitHub.

Building it required me to think about authentication, API limits, structured data, repository state, conflicting changes, auditability, failure handling and where human judgement belongs in an AI-assisted workflow.

It brings together the parts of technical work I enjoy most: investigating how systems behave, understanding the customer impact, designing practical workflows and turning complex changes into clear information people can trust.
